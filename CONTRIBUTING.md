
🧩 CONTRIBUTING.md — Version personnalisée pour tes projets GitHub

`markdown

🤝 Guide de Contribution

Merci de votre intérêt pour contribuer à ce projet.  
Ce dépôt suit une organisation professionnelle, modulaire et orientée qualité.  
Veuillez lire attentivement les règles ci‑dessous avant de proposer une contribution.

---

🧭 1. Comment contribuer

Vous pouvez contribuer de plusieurs manières :

- 🐛 Rapporter un bug via Issues → Bug Report
- 💡 Proposer une amélioration via Issues → Feature Request
- 🔧 Soumettre un Pull Request
- 📚 Améliorer la documentation
- 🎨 Proposer des optimisations UI/UX ou design

---

🌱 2. Branches et workflow Git

- La branche principale est : main
- Pour toute contribution, créez une branche dédiée :

`
git checkout -b feature/nom-de-la-feature
`

- Les noms de branches doivent suivre ce format :

| Type | Format |
|------|--------|
| Feature | feature/nom |
| Fix | fix/nom |
| Docs | docs/nom |
| Refactor | refactor/nom |

---

🧪 3. Pull Requests

Avant d’ouvrir un PR :

- Vérifiez que votre code compile
- Vérifiez que les fichiers inutiles ne sont pas commit
- Vérifiez que votre PR est claire, concise et documentée

✔ Checklist PR

- [ ] Le code respecte la structure du projet  
- [ ] Le code est lisible et commenté si nécessaire  
- [ ] Le PR décrit clairement le changement  
- [ ] Aucun fichier inutile (cache, build, node_modules…)  
- [ ] Le workflow GitHub Actions passe sans erreur  

🔥 Règle YOLO (optionnelle)

Pour les mainteneurs :  
Vous pouvez merger un PR sans review uniquement pour les correctifs mineurs.

---

📁 4. Structure du projet

Merci de respecter l’organisation suivante :

`
/src        → Code source
/docs       → Documentation
/design     → Identité visuelle
/.github    → Workflows, templates, config
/assets     → Images, icônes, médias
`

---

🧰 5. Normes de code

- Utiliser des noms explicites
- Éviter les duplications
- Préférer la modularité
- Documenter les fonctions complexes
- Respecter les conventions du langage utilisé

---

📝 6. Issues

Avant d’ouvrir une issue :

1. Vérifiez qu’elle n’existe pas déjà  
2. Utilisez le bon template  
3. Fournissez des détails reproductibles  
4. Ajoutez des captures si nécessaire  

---

🧑‍💻 7. Discussions

Les discussions sont ouvertes pour :

- Brainstorming  
- Questions générales  
- Propositions de roadmap  
- Retours utilisateurs  

---

⭐ 8. Merci

Chaque contribution compte.  
Merci de participer à l’amélioration de ce projet et de soutenir son évolution.
`
